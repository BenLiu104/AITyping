> 按住說話 → 轉寫 → AI 整理 → 出現在 textarea → 一鍵 copy / share。
>
> 等流程成熟，再考慮 native iOS keyboard extension。

像先在夜裡點一盞小燈，不急著建整座城市。

## 2. 你的流程逐步評估

你原本設計：

```text
按下語音輸入按鍵
→ 建立 Live API session
→ mic audio chunk 持續送入 WebSocket
→ Live API 回傳 inputTranscription
→ app 暫存 transcript
→ 放開語音輸入按鍵
→ transcript 送給 gemini-3.1-flash-lite 做 final cleanup
```

這個流程本身是合理的。我要微調成 production-friendly 版本：

```text

→ 建立 Gemini Live API WebSocket session

→ 解析 inputTranscription

→ textarea 顯示即時草稿

使用者放開 mic
→ 停止 mic tracks
→ 關閉 / flush Live session
→ 將累積 transcript 送到後端 cleanup endpoint
→ 後端呼叫 gemini-3.1-flash-lite
→ 回傳 cleaned text
→ 插入 textarea
→ 顯示 copy / share / append / replace 選項
```

## 3. Gemini Live API 部分：可行，而且方向正確

Google Gemini Live API 官方文件目前描述它是：

- stateful WebSocket API
- 支援 audio / image / text streaming
- audio input format 是 raw 16-bit PCM
- native input sample rate 是 16kHz
- 可以啟用 `inputAudioTranscription`
- browser client production 建議用 ephemeral token

你的用途不是要 Gemini 跟人對話，而是要低延遲 transcription。這其實很適合 Live API。

### 3.1 WebSocket endpoint 概念

官方 raw WebSocket endpoint 類似：

```text
wss://generativelanguage.googleapis.com/ws/google.ai.generativelanguage.v1beta.GenerativeService.BidiGenerateContent?key=API_KEY
```

但 production **不要把 API key 放前端**。前端可以被任何人打開 DevTools 看見 key。
更好的做法是：

```text
Browser → 你的 VPS → 產生 ephemeral token → Browser 直接連 Gemini Live API
```

也就是：

```text
iPhone Safari
   ↓ request token
VPS backend
   ↓ uses real GEMINI_API_KEY
Gemini auth token endpoint
   ↓ returns short-lived token
iPhone Safari
   ↓ direct WebSocket with ephemeral token
Gemini Live API
```

這樣 audio 不需要經過你的 VPS，latency 會低很多，VPS 壓力也小。

## 4. iPhone Web App 收音：可行，但關鍵是不要靠 MediaRecorder

iOS Safari 支援 microphone access，但這裡有一個很重要的技術分岔。

### 方案 A：MediaRecorder

```js
const recorder = new MediaRecorder(stream)
recorder.start(250)
```

這種做法簡單，但在 iOS Safari 上通常會產生 `audio/mp4` / AAC 類型資料。

問題是 Gemini Live API 要的是：

```text
raw little-endian 16-bit PCM audio
16kHz recommended
mimeType: audio/pcm;rate=16000
```

所以 MediaRecorder 產出的 AAC/MP4 chunk **不是最適合直接餵 Live API**。你還要 decode / transcode，會增加複雜度與延遲。

### 方案 B：Web Audio + AudioWorklet

這才是推薦路線。

```text
getUserMedia()
→ AudioContext
→ MediaStreamSource
→ AudioWorkletNode
→ 收到 Float32 PCM frame
→ resample to 16kHz
→ convert Float32 to Int16 PCM
→ base64
→ websocket.send({ realtimeInput: { audio: ... }})
```

這個方案比較「硬核」，但最乾淨、最低延遲、最接近 Live API 原生格式。

## 5. iOS Safari / PWA 的限制

Web app 可行，但要接受幾個邊界。

### 5.1 必須 HTTPS

`getUserMedia()` 只在 secure context 可用。
所以你的 VPS 必須有：

- domain
- HTTPS certificate
- reverse proxy，例如 Caddy / Nginx / Cloudflare Tunnel

localhost 在開發機可以，但 iPhone 實機測試基本上要 HTTPS。

### 5.2 使用者必須主動觸發 microphone

iOS 不喜歡自動播放 / 自動收音。
所以流程最好是：

```text
使用者 touchstart / pointerdown mic button
→ 才 request permission / start AudioContext
```

不要頁面載入就偷偷開 mic。Safari 會擋，使用者也會不舒服。

### 5.3 背景收音不可靠

如果使用者切 app、鎖屏、Safari 退到背景，web app 收音很可能中斷。
所以你的 UX 應該假設：

```text
使用者打開頁面
→ 短句 / 中短段語音輸入
→ 結束
```

不要假設它能像 native dictation 一樣長時間背景運作。

### 5.4 不能直接把文字塞到其他 app

這是最大的產品限制。

Web app 可以：
- 插入自己頁面上的 textarea
- 自動 select text
- 使用 Clipboard API 在使用者動作下 copy
- 呼叫 share sheet
- 讓使用者手動 paste 到別處

Web app 不可以：
- 控制 iOS 其他 app 的輸入框
- 變成全系統 keyboard
- 在 Telegram/WhatsApp/Notes 裡直接插字
- 模擬系統 paste 到別的 app

所以如果你想要「任何地方都能按 mic，然後文字直接進當前 input」，那最終還是要 native keyboard extension。

## 6. iOS Keyboard App：理想，但不是現在最舒服的 MVP

你說得很對：**最理想是 iOS keyboard app，但 VPS 開發環境比較難實現。**

### 6.1 為什麼 keyboard app 理想？

因為它可以出現在任何 text field：

```text
Telegram input
Notes
Safari search bar
Gmail
WhatsApp
任何支援第三方鍵盤的 app
```

它可以透過 `textDocumentProxy.insertText(...)` 把文字插進當前輸入框。這是 web app 做不到的。

### 6.2 但 custom keyboard extension 有不少限制

Apple 對 keyboard extension 很嚴格，因為 keyboard 可以看見使用者輸入的敏感資訊。常見限制包括：

| 限制 | 影響 |
|---|---|
| Secure text field 不能用第三方鍵盤 | 密碼欄會切回系統鍵盤 |
| 某些 app 可以拒絕第三方 keyboard | 銀行、醫療、隱私 app 常見 |
| keyboard 必須有切換鍵盤按鈕 | 例如 globe key |
| 網路功能通常需要 Full Access | 使用者要手動開 |
| Extension 記憶體 / 執行環境受限 | 不適合重型 AI / 音訊處理 |
| mic / audio session 在 extension 內很麻煩 | 可靠性和 App Review 都要小心 |
| 必須用 Xcode / Apple Developer tooling | VPS-only 不舒服 |

Apple 舊版官方文件甚至明確提到 custom keyboard 無法使用 dictation input，因為 extension microphone access 受限。實務上「keyboard extension 裡做穩定長時間錄音 + 網路串流」仍然是高摩擦路線。

| Backend | FastAPI 或 Node/Express |

## 9. Live API setup 概念

WebSocket 第一個 message 要 setup：

```json
{
  "setup": {
    "model": "models/gemini-3.1-flash-live-preview",
    "responseModalities": ["AUDIO"],
    "inputAudioTranscription": {},
    "systemInstruction": {
      "parts": [
        { "text": "Transcribe the user's speech accurately. Do not answer conversationally." }
      ]
    }
  }
}
```

核心是 `inputAudioTranscription: {}`。然後送 audio：

```json
{
  "realtimeInput": {
    "audio": {
      "data": "BASE64_PCM16_AUDIO",
      "mimeType": "audio/pcm;rate=16000"
    }
  }
}
```

收到 server message 後解析 transcription parts。

## 10. Cleanup prompt 建議

語音輸入常有：重複字、停頓詞、口語碎片、中英混合、標點缺失、粵語 / 國語 / 英文混雜。cleanup endpoint 接：

```json
{ "rawTranscript": "...", "mode": "message", "language": "zh-Hant", "style": "natural" }
```

Prompt 可以是：

```text
你是一個語音輸入文字整理器。
任務：
1. 保留使用者原意。
2. 修正語音轉寫錯字。
3. 移除無意義停頓詞，例如「呃」「嗯」「就是就是」。
4. 補上自然標點。
5. 不要加入原文沒有的新資訊。
6. 如果內容是中英混合，保留自然中英混合。
7. 輸出最終可直接貼上的文字，不要解釋。
輸入：
{{rawTranscript}}
```

不同模式：訊息（自然簡潔適合聊天）/ Email（禮貌清楚可直接寄出）/ TODO（每項動詞開頭）/ Prompt（清楚具體可直接給 AI）。真正讓人上癮的不是 transcription，而是「講得亂，出來很乾淨」。

## 11. 前端 audio pipeline 重點

**Mic permission：**

```js
const stream = await navigator.mediaDevices.getUserMedia({
  audio: { echoCancellation: true, noiseSuppression: true, autoGainControl: true }
})
```

**AudioContext：** `new AudioContext({ sampleRate: 16000 })` — 注意 Safari 可能不尊重指定 sample rate（常給 44100 / 48000），要在 JS 裡自己 resample。

**Float32 to PCM16：**

```js
function floatTo16BitPCM(float32Array) {
  const buffer = new ArrayBuffer(float32Array.length * 2)
  const view = new DataView(buffer)
  for (let i = 0; i < float32Array.length; i++) {
    let s = Math.max(-1, Math.min(1, float32Array[i]))
    view.setInt16(i * 2, s < 0 ? s * 0x8000 : s * 0x7fff, true)
  }
  return buffer
}
```

**Chunk size：** 建議每次送 20ms–100ms。以 100ms / 16kHz / PCM16 mono = 16000 × 0.1 × 2 = 3200 bytes，base64 後約 4.3KB，很合理。

## 13. 風險與解法

| 風險 | 影響 | 解法 |
|---|---|---|
| iOS Safari audio permission 麻煩 | 首次體驗有阻力 | 首次 onboarding 清楚說明 |
| AudioWorklet / resampling 實作複雜 | 開發時間增加 | 先做最小 PCM pipeline |
| Live API preview 狀態 | API 可能變動 | 封裝 adapter，不要散落全 app |
| WebSocket 中斷 | transcript lost | local buffer + reconnect |
| partial transcript 不穩 | 顯示跳動 | 分開 partial / committed transcript |
| API key 暴露 | 安全問題 | 必須用 ephemeral token |
| 無法貼到其他 app | 產品限制 | PWA + copy/share；長期做 keyboard extension |
| cleanup 改錯意思 | 使用者不信任 | 提供 raw transcript toggle / undo |

## 14. MVP 開發順序

1. 建立基本 web UI（textarea、mic button、status、copy、mode selector，先不接 Gemini）
2. 確認 iPhone Safari mic 可用（`getUserMedia({ audio: true })` + 音量 meter）
3. AudioWorklet 取得 PCM（debug panel 顯示 sampleRate / chunk samples / 轉換後 bytes）
4. 接 Live API WebSocket（先 desktop Chrome 再 iPhone Safari，先不 cleanup 只顯示 transcript）
5. 加 cleanup endpoint（`POST /api/cleanup`，用 `gemini-3.1-flash-lite`）
6. iPhone 真機測試（Safari / PWA、Wi-Fi / 5G、短句 / 長句、中英混合 / 繁中 / 粵語口語）
7. 打磨 UX（auto-copy、vibration feedback、release-to-send、cancel gesture、history、prompt presets）

## 15. Web App vs iOS Keyboard 的取捨

**現在做 Web App：** VPS 開發舒服、不需 Mac / Xcode、可快速驗證 Gemini Live API pipeline、可快速打磨 cleanup prompt、可直接部署 HTTPS 給 iPhone 測、失敗成本低。

**暫時不要先做 iOS Keyboard：** native extension 環境麻煩、App Review / privacy / Full Access 都是坑、mic + WebSocket 在 extension 裡風險更高、還沒驗證產品 flow 前不值得跳進蘋果花園的荊棘。

---

## 16. 最推薦的第一版規格

> **一個 iPhone-first PWA：智能語音草稿板。**

MVP 必須有：iPhone Safari 可用、按住說話、即時 transcript、放開後 cleanup、輸出到 textarea、一鍵 copy、繁中 / 英文 / 中英混合、訊息 / email / prompt / TODO 模式。

MVP 不做：登入、付款、native keyboard、長時間背景錄音、自動貼到其他 app、複雜多人同步。

---

## 17. 最後判斷

技術上非常接近可做，用 VPS 開發 web app 是合理路線。但要記住：

> **Web App 可以做「智能語音輸入器」，不能做「系統級輸入法」。**

所以產品定位先是 `AI voice drafting PWA`，不是 `iOS system keyboard replacement`。等 PWA 真的好用、prompt cleanup 真的準、你每天願意打開它，再投資 native iOS keyboard app —— 那時候你已經有了核心引擎，只是換一個外殼。這樣走，風險最小，路也最亮。
