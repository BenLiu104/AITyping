## 可行性分析：iOS 智能語音輸入 Web App

### 1️⃣ 方案一：Web App（Safari on iOS）

**核心技術棧：**

| 層面 | 技術 | iOS Safari 支援度 |
|------|------|-------------------|
| 麥克風擷取 | `navigator.mediaDevices.getUserMedia()` | ✅ iOS 14.3+ 完整支援 |
| 音訊串流 | `MediaRecorder` / `AudioContext` + `WebSocket` | ✅ 已支援 |
| 即時語音轉文字 | Gemini Live API (WebSocket) | ✅ 純 JS + WS 即可 |
| 最終清理 | Gemini API (REST) | ✅ |

**你的流程，在 Web App 裡可以這樣實作：**

```
用戶按下按鈕 (touchstart)
  → navigator.mediaDevices.getUserMedia({ audio: true })
  → AudioContext.createMediaStreamSource(stream)
  → 將 PCM 音頻 chunk 透過 WebSocket 送入 Gemini Live API
  → Live API 回傳 inputTranscription (即時文字)
  → 顯示在 textarea 裡，邊說邊出字
用戶放開按鈕 (touchend)
  → 關閉 WebSocket
  → 把整段文字送給 gemini-3.1-flash-lite (REST API) 做清理
  → 結果貼回 textarea
```

**關鍵優勢：**
- **不需要 Mac / Xcode**，你的 VPS 就能全權開發
- 部署極簡單 —— 一個 nginx 或 Caddy 靜態站 + API key 就夠
- iOS Safari 對 `getUserMedia`、`MediaRecorder`、`WebSocket` 的支援已經很成熟
- Gemini Live API 有官方 JavaScript WebSocket 範例，前端直接連即可

**需要注意的限制：**

| 問題 | 說明 | 解法 |
|------|------|------|
| API Key 暴露 | 前端 JS 直接帶 key 連 Gemini，key 會被看到 | 加一層輕量後端 proxy（Node/Python），前端連自己的 server，server 帶 key 連 Gemini |
| iOS 背景限制 | Safari 切到背景會暫停 audio capture | 這是設計使然，你的 app 是前台操作，不影響 |
| 音頻格式轉換 | Gemini Live API 要 16kHz PCM 16-bit LE | 用 `AudioContext` + `ScriptProcessorNode` / `AudioWorklet` 做 resample + format 轉換 |
| 延遲 | WebSocket + Gemini 處理有 ~200-500ms 延遲 | 對「邊說邊出字」來說完全可以接受 |

**關於 Web Speech API：**
Safari 的 `SpeechRecognition` API **不支援**。Apple 只在 Safari 實作了 `SpeechSynthesis`（TTS），沒有做 STT。所以不能走 Chrome 那條 `webkitSpeechRecognition` 的路線。但這不是問題——我們直接用 Gemini Live API 做語音辨識，效果反而更好（多語言、更準確）。

### 2️⃣ 方案二：iOS Keyboard Extension（理想但困難）

**為什麼理想：**
- 可以直接插入任何 text field（WhatsApp、Notes、Mail…）
- 體驗最流暢，像原生鍵盤一樣

**為什麼難：**
| 門檻 | 說明 |
|------|------|
| **必須有 Mac** | Keyboard Extension 必須用 Xcode 編譯、簽署 |
| **Apple Developer Program** | $99/年，需要 Provisioning Profile |
| **Swift / Objective-C** | 不能用 Python 或 JS 開發 |
| **VPS 完全幫不上忙** | 開發和簽署只能在 macOS 上做 |

**結論：沒有 Mac 的話，這條路幾乎走不通。**（雖然有一些 circumvention 方法如 AltStore，但體驗和穩定性差很多。）

---

### 3️⃣ 我的建議：先做 Web App，再考慮 PWA

**Phase 1 — MVP（1-2 天可完成原型）：**

```
前端：單頁 HTML + JS
  ├── 麥克風按鈕（長按說話）
  ├── WebSocket 連 Gemini Live API
  ├── 即時顯示 transcription
  └── 鬆手後調 REST API 做 final cleanup

後端 Proxy（部署在 VPS）：
  ├── 保護 API key
  ├── 可做 rate limiting
  └── 未來加 auth / 計費
```

**Phase 2 — PWA 包裝：**
- 加 `manifest.json` + service worker
- 用戶可以 "Add to Home Screen"，看起來像原生 app
- 全螢幕、無瀏覽器地址列

**Phase 3 — 如果未來有 Mac：**
- 再把 Keyboard Extension 補上，作為進階選項

### 總結

| 維度 | Web App | Keyboard Extension |
|------|---------|-------------------|
| 可行性 | ✅ 高 | ❌ 需要 Mac |
| 開發速度 | 快（JS 為主） | 慢（Swift + Xcode） |
| 部署難度 | 極低（VPS 即可） | 需 Apple Developer |
| 用戶體驗 | 好（PWA 更佳） | 最佳 |
| 跨平台 | ✅ iOS + Android | ❌ 僅 iOS |

**結論：Web App 完全可行，而且是你目前環境下最實際的選擇。** 先用 Gemini Live API + WebSocket 做即時語音轉文字，鬆手後用 Gemini REST API 做文字清理，整個流程在 iOS Safari 上都能跑。
