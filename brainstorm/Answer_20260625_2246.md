## 🔍 可行性分析：iOS Web App + Gemini Live API 智能語音輸入

### 一、核心技術路線驗證

你的構想是可行的，但有幾個關鍵技術點需要釐清：

#### ✅ 可行的部分

| 技術 | iOS Safari 支援 | 備註 |
|------|----------------|------|
| `getUserMedia()` 麥克風 | ✅ iOS 14.3+ | Safari 已完整支援 |
| `MediaRecorder` API | ✅ iOS 14.3+ | 可錄製 audio/webm |
| `Web Audio API` + `AudioContext` | ✅ iOS Safari | 可取原始 PCM 資料 |
| WebSocket | ✅ 全平台 | 標準支援 |
| Gemini Live API (WebSocket) | ✅ 公開可用 | `gemini-3.1-flash-live-preview` |

#### ⚠️ 需要注意的部分

**Gemini Live API 要求的音訊格式是 16kHz raw PCM (16-bit, little-endian)**，但手機麥克風原始輸出通常是 44.1kHz 或 48kHz。這意味著你需要在瀏覽器端做 **sample rate conversion**。

### 二、兩種實作架構比較

#### 方案 A：純前端（直接從瀏覽器連 Gemini Live API）

```
iPhone Safari → getUserMedia() → Web Audio API → Resample to 16kHz PCM
                                            ↓
                                    WebSocket → Gemini Live API
                                            ↓
                                    inputTranscription → 顯示
                                            ↓
                                    (放開按鈕) → gemini-3.1-flash-lite cleanup
                                            ↓
                                    貼到 text area
```

**優點：**
- 不需要後端伺服器，成本最低
- 延遲最低（直連 Google）
- 架構簡單

**缺點：**
- API Key 會暴露在前端（可以用 Cloudflare Worker 做 proxy 解決）
- 瀏覽器端做 resampling 需要寫 Web Audio API node 轉換
- iOS Safari 對 `MediaRecorder` 的 chunk 間隔控制不如 desktop 精細

#### 方案 B：前端 + 輕量後端 Proxy（推薦）

```
iPhone Safari → getUserMedia() → Web Audio API → Resample to 16kHz PCM
                                            ↓
                                    WebSocket → Cloudflare Worker / VPS
                                            ↓
                                    Proxy → Gemini Live API
                                            ↓
                                    inputTranscription → 回傳前端
                                            ↓
                                    Cleanup → gemini-3.1-flash-lite (後端)
                                            ↓
                                    回傳結果 → 前端貼上
```

**優點：**
- API Key 安全（存在 VPS 上）
- 後端可以做 cleanup 邏輯
- 可以加 queue、rate limit

**缺點：**
- 需要一個輕量 proxy（你的 VPS 就能跑）
- 多一跳網路延遲（約 50-100ms）

### 三、iOS Safari 關鍵坑點

#### 1. 麥克風權限
- 必須是 **HTTPS** 頁面（或 localhost）
- 用戶需手動授權麥克風
- PWA（加到主畫面）體驗更好，但 standard HTTPS 網頁也夠用

#### 2. 音訊采樣率轉換
```javascript
// 核心程式碼概念
const audioContext = new AudioContext();
const source = audioContext.createMediaStreamSource(stream);

// 用 ScriptProcessorNode 或 AudioWorklet 做 resample
const resampler = audioContext.createScriptProcessor(4096, 1, 1);
resampler.onaudioprocess = (e) => {
  // 44.1kHz → 16kHz downsample
  const inputData = e.inputBuffer.getChannelData(0);
  const pcm16k = downsample(inputData, audioContext.sampleRate, 16000);
  // base64 encode → send via WebSocket
  ws.send(JSON.stringify({
    realtimeInput: {
      audio: btoa(String.fromCharCode(...pcm16k))
    }
  }));
};
```

#### 3. 按住說話 vs 連續錄音
- 你描述的是「**按住說話、放開停止**」的邏輯
- 這需要監聽 `pointerdown` / `pointerup` 事件來控制 WebSocket 的 audio 傳送
- 放開後再調一次 Gemini cleanup API

#### 4. iOS Safari 的特殊行為
- `MediaRecorder` 在 iOS 上支援 `audio/webm`，但 chunk 最小間隔建議 ≥ 100ms
- 如果用 `AudioContext` 直接取 PCM 更可靠（不依賴 MediaRecorder）
- Safari 會自動壓縮音訊，建議用 `AudioContext` 拿 raw PCM

### 四、你的流程優化建議

你原本的流程：
> mic → Live API → inputTranscription → 放開 → cleanup → 貼上

我建議的微調：

1. **按住時**：Live API 的 `inputTranscription` 本身就是即時轉寫，已經很準了
2. **放開後的 cleanup**：不是必須的。你可以直接把 `inputTranscription` 的結果貼上去
3. **如果想提升品質**：cleanup 可以用 system prompt 引導 Gemini 做「語句潤飾、斷句修正、繁簡統一」，而不是完全重新生成
4. **建議的 cleanup prompt**：
   ```
   將以下語音轉寫結果整理成通順的繁體中文段落。
   保留原意，修正語病和斷句，不要添加額外內容。
   只輸出整理後的文字，不要解釋。

   原文：{transcript}
   ```

### 五、iOS Keyboard Extension vs Web App

| 維度 | Web App (PWA) | iOS Keyboard Extension |
|------|--------------|----------------------|
| 開發難度 | ⭐⭐ 低 | ⭐⭐⭐⭐⭐ 極高 |
| 上架 | 無需 App Store | 需要 Apple Developer 帳號 ($99/年) |
| 麥克風權限 | 每次開啟頁面請求 | 首次安裝時請求 |
| 使用者體驗 | 加到主畫面後接近原生 | 鍵盤內直接觸發，最流暢 |
| 維護更新 | 即時（改網頁就好） | 需要重新提交 App Store 審核 |
| VPS 開發 | ✅ 完全可行 | ❌ 需要 Mac + Xcode |

**結論：** Web App 是完全可行的起步方案。等驗證了 product-market fit，再考慮做 iOS Keyboard Extension 或 native app。

### 六、推薦的技術棧

```
前端：
  - Vanilla JS 或 Svelte（極簡、無框架負擔）
  - Web Audio API（取 mic PCM）
  - WebSocket（連 Gemini Live API）
  - Service Worker（可做 PWA cache）

後端 Proxy（VPS 上跑）：
  - Node.js + ws 或 Python + FastAPI
  - 只做 WebSocket proxy + API key 保護
  - 也可以直接用 Cloudflare Workers（免費額度夠用）

Cleanup：
  - 同一個 Gemini API，用 gemini-3.1-flash-lite
  - HTTP REST endpoint（不需要 Live API）
```
