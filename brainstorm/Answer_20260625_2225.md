## TL;DR

| 方案 | 可行性 | 卡在哪 |
|---|---|---|
| **Web app (PWA) 做 push-to-talk 語音輸入** | ✅ 完全可行 | 只需 HTTPS + 一點 AudioWorklet 工夫 |
| **真正的 iOS 系統鍵盤 (keyboard extension) 內建即時麥克風** | ❌ 此路不通 | **不是 VPS 問題，是 Apple 從 OS 層封死** |

關鍵誤解先拆掉：你以為 keyboard app 做不到是因為「VPS 開發環境難實現」。其實**就算你有 Mac + Xcode 也做不到即時麥克風**——Apple 規定**任何 app extension（包括自訂鍵盤）都不能存取麥克風**，理由是 iOS 沒辦法清楚提示「手機正在聽你講話」。([MacStories 證實，iOS 27 仍然如此](https://www.macstories.net/notes/on-the-limitations-of-ios-custom-keyboards))

---

## 為什麼 keyboard app 這條路要放棄

1. **Apple 硬限制**：custom keyboard extension 拿不到 mic。Stack Overflow 上有人實測，模擬器能錄、真機完全錄不到、也不報錯。
2. **市面上的「語音鍵盤」是怎麼做的？** 它們在作弊：
   - **Gboard 模式**：長按空白鍵 → **跳出鍵盤、短暫打開它自己的主 app**（主 app 才有 mic）→ 在主 app 錄音轉文字 → 複製回去 → 跳回原本的 app。體驗割裂。
   - Willow / Spokenly 等宣稱「鍵盤內聽寫」，靠 Full Access 鑽空子，**脆弱、無官方文件、Apple 隨時可封**。
3. **無論哪種，都需要原生 Swift app + Mac + Xcode + Apple Developer 帳號（$99/年）**。你的 VPS（Linux ARM64）編譯不出 iOS app。**所以這條路對你是雙重封死。**

> 一句話：keyboard 的夢想 = 「語音輸入能貼進**任何** app（WhatsApp、Notes…）」。這個系統級能力 Apple 不給 web，只給原生鍵盤，而原生鍵盤又不給 mic。死局。

## Web app 為什麼可行 — 逐步對照你的 flow

你的 flow 在 iOS Safari 的真實情況：

| 你的步驟 | iOS Safari 現實 | 備註 |
|---|---|---|
| 按下按鍵 → 建立 Live session | ✅ 可以，但 `AudioContext` 必須在**觸控事件裡** `.resume()` | iOS 規定要 user gesture 才能開麥 |
| mic chunk 持續送 WebSocket | ✅ getUserMedia 前景可用 | **背景會被 suspend**（[WebKit bug 226620](https://bugs.webkit.org/show_bug.cgi?id=226620)） → 但 push-to-talk 你一直盯著螢幕、按著鈕 = 前景，**沒問題** |
| Live API 回傳 inputTranscription | ✅ 官方支援 client 直連 | 需在 setup config 開 `inputAudioTranscription` |
| 暫存 transcript | ✅ 純前端 | 小事 |
| 放開 → 送 flash-lite cleanup | ✅ 走你 VPS 後端 | 把 API key 藏在後端，別放前端 |
| 貼上 text input area | ✅ **貼回你自己 app 的 `<textarea>`** | ⚠️ 只能貼回自己頁面，**不能貼到別的 app**（那是 keyboard 才有的能力） |

**直連 Gemini Live 的官方做法**：你 VPS 後端用 `google-genai` SDK 簽發 **ephemeral token（短效）**，瀏覽器拿 token **直接 WebSocket 連** `generativelanguage.googleapis.com`，API key 永不外洩。Google 官方有完整範例：[gemini-live-ephemeral-tokens-websocket](https://github.com/google-gemini/gemini-live-api-examples/blob/main/gemini-live-ephemeral-tokens-websocket/README.md)。

## 一個重要的工程建議：先別上 Live API

你描述的 Live streaming 是「理想版」，但它是**最重的做法**（WebSocket + PCM 降頻 worklet + token 管理）。同樣的 push-to-talk 體驗，有更省力的路：

| 方案 | UX | 工程量 | 建議 |
|---|---|---|---|
| **A. Live API streaming**（你的計劃） | 講的同時即時看到字 | 高：WebSocket + AudioWorklet 降到 16kHz/16-bit PCM + ephemeral token | 想要「邊講邊出字」才上 |
| **B. Batch 錄音上傳**（建議當 MVP） | 放開後 ~1–2 秒出全文 | 低：`MediaRecorder` 錄一段 → 放開後整段丟給 Gemini 轉寫+cleanup | **先做這個** |
| C. Web Speech API | 不穩、限時長、品質普通 | 低 | 不建議當主力 |

**我的建議：先用 B 做出能用的 MVP**（一個按鈕、一個 textarea、放開就出乾淨文字），確認整條鏈路通了、cleanup prompt 調好了，**再升級成 A 加上即時 partial transcript**。先 streaming 會卡在 iOS 的 AudioWorklet 降頻細節上耗掉你大半時間。

## 你 VPS 上要跑什麼

```
iPhone Safari (PWA, 前景 push-to-talk)
        │  HTTPS（必須！getUserMedia 不給 http://IP）
        ▼
你的 VPS (161.153.57.166, Docker)
 ├── 靜態前端  index.html + worklet.js
 ├── POST /token    → google-genai 簽 ephemeral token（Live 版才需要）
 └── POST /cleanup  → 呼叫 flash-lite，API key 留在後端
        │
        ▼（A 版才有）瀏覽器拿 token 直連
  generativelanguage.googleapis.com  (Live WebSocket)
```

**HTTPS 是硬需求**（iOS 麥克風只在 secure context 給）。你已經有 **cloudflare-tunnel** 的技能，直接拿來把 VPS 的 80 端口包成 `https://你的域名` 最快。

## iOS Safari 必踩的坑（checklist）

- [ ] `AudioContext` 預設 suspended → **在按鈕的 touch handler 裡 `resume()`**
- [ ] iOS sample rate 常鎖 48000 → Live 版要在 AudioWorklet **降頻到 16kHz、轉 Int16 PCM**
- [ ] 必須 **HTTPS**（Cloudflare Tunnel / Caddy+Let's Encrypt）
- [ ] 背景會斷麥 → push-to-talk 前景使用就避開了
- [ ] 「加入主畫面」的 standalone PWA 偶有麥克風 regression → 先用**普通 Safari 分頁**測穩了再說
- [ ] 確認 `gemini-3.1-flash-lite` 這個 model id 是否為現行名稱（模型名常改，上線前對一次官方 model list）
